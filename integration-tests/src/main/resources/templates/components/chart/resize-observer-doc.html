Why resize handling matters

ECharts measures its container once at init. Without a resize hook it stays at that initial size forever — a chart inside a sidebar that opens later, a dashboard tile that re-flows when the parent grid changes columns, or any chart embedded in an HTMX-swapped fragment will all render at the wrong dimensions.

What phaChart does

On init(), the Alpine factory registers a single resize handler on window that calls chart.resize(). On destroy() it removes the handler and disposes the ECharts instance so there's no leak.

This covers the common viewport-resize case. To opt out, pass {|phaChart({ option, autoResize: false })|}; you keep getChart().resize() as a public method you can call manually from elsewhere in the page.

When the parent re-flows without a window resize

A window listener won't catch container-only resizes — e.g. a CSS-grid cell that grows because a sibling tile collapses, or an HTMX swap that resizes the parent flexbox. For those, attach a ResizeObserver to the wrapper:

JavaScript 
document.addEventListener('alpine:init', () => {
  Alpine.directive('chart-resize', (el, { expression }, { evaluateLater }) => {
    let target = el.querySelector('.pha-c-chart');
    if (!target || !target.__phaChart) return;
    let ro = new ResizeObserver(() => target.__phaChart.resize());
    ro.observe(el);
  });
});

Then add x-chart-resize to the parent container — any dimension change on the parent triggers a chart re-layout.

HTMX swaps clean up cleanly

When HTMX replaces a wrapper via outerHTML, Alpine's destroy() runs on the old node — which disposes the old ECharts instance — and the new node's init() creates a fresh one. There's nothing to wire up.
